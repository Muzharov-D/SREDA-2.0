# СРЕДА — Приоритеты переделки демо

> Как перевести текущее демо (`assets/app.js` + оверлеи) на модель из [concept-interface-v1.md](../concept-interface-v1.md) и [CONCEPT-SCREENS-v1.md](CONCEPT-SCREENS-v1.md).
> Принцип: **не переписывать с нуля, а собрать первый экран заново и понизить остальное по приоритету.** Перегруз лечится иерархией, а не удалением функций.

---

## Диагноз: откуда «навалено»

`renderStage()` (app.js:1311) роутит в **~30 экранов-архетипов**:
`pulse, dpulse:, company, organism, dashboard, router, audit, project, roadmap, core, roles, power, cockpit, battle, marketplace/lib:, team:, flow:, flowx, federation, contracts, arbitration, economy, domain:, studio, forge, modules, talent, workers, aibudget, asst:` + профили.

Проблема не в том, что их много, а в том, что **они равноправны и рисовались «от сценария», а не выводятся из модели**. У рядового пользователя нет единой точки входа «мой день» — есть меню из десятков экранов.

## Целевая карта: 3 высоты + сквозной помощник

Всё сводится к **одному Пульсу на трёх высотах** + профили/библиотека как вложенные виды:

| Новая сущность | Что переиспользуем из кода | Что делаем |
|---|---|---|
| **Пульс «Я»** (Экран 1) | — (в демо НЕТ личного пульса) | **строим с нуля** — это ядро и главный пробел |
| **Пульс «Отдел»** | `renderDeptPulse` (app.js:971), `dpulse:` | переосмыслить: передачи + ждёт-отдел + загрузка |
| **Пульс «Компания»** | `renderPulse`/`renderOrganism` (1335/1375), `neural.js`, `organism.js` | пере-заземлить: узел = реальный ЦС, синапс = реальная передача |
| **Мой штат** (рельс) | `renderTeam` (317), `worker:`/`person:` профили (`rpg.js`) | ужать в левый рельс + карточка по клику |
| **Личный помощник** | `renderAssistant` (494), `asst:`, `kam-assist.js` | из экрана → в **сквозную строку**, контекст = текущий экран |
| **Конструктор+эскалация** (Экран 2) | частично `forge.js`/`renderForge` | новый флоу «чего не хватает → каталог / эскалация» |
| **Кандидаты из Zoom** (Экран 3) | `kam-assist.js` | новый экран разбора протокола |
| **Онбординг** (Экран 5) | `renderLibrary`/`marketplace` (2303), `renderRoles` (2565) | флоу ТЗ → штат из библиотеки |

## Фазы

### Фаза 0 — Каркас высот (1 шаг, обратимо)
- Ввести переключатель `[ Я | Отдел | Компания ]` в топбар с гейтом роли (Компания = Оркестратор).
- Замапить существующее: `Компания → renderPulse/organism`, `Отдел → renderDeptPulse`. «Я» пока заглушка.
- **Результат:** появляется единая ось навигации вместо плоского меню.

### Фаза 1 — Ядро «Пульс Я» + разгрузка (главный удар по перегрузу)
- Построить **Пульс «Я»** (Экран 1): «Ждёт меня» (§7.2) + встречи + ЦС в работе + предложения. Это то, чего в демо нет.
- **Свернуть навигацию по умолчанию:** рядовой пользователь видит только Пульс + Мой штат. Всё остальное — не в дефолтном меню.
- **Понизить/убрать из дефолта** (перенести под Оркестратора/Админа или в архив демо): `battle, power, core, federation, contracts, arbitration, economy, domain:, studio, modules, flowx, talent`. Это спекулятивные/продвинутые экраны — они множат «навалено» на входе.
- **Результат:** первый экран показывает только то, что требует пользователя сегодня.

### Фаза 2 — Единая грамматика
- **Подсветка = 2 цвета точек участия** (🔴 блок на тебе / 🟡 вопрос к тебе) во всех Пульсах. Убрать «срочность/декор».
- **Передачи получают смысл:** в `organism.js`/`neural.js` узел = реальный ЦС/человек из данных, синапс = реальная передача (FLOWS), гейт = §7.2. (Закрывает старые жалобы из `concept-redesign.md`.)
- **Помощник → сквозной:** `renderAssistant` из полноэкранного в нижнюю строку, контекст = текущий экран.
- **Результат:** все экраны читаются одним языком.

### Фаза 3 — Дифференциаторы
- **Кандидаты из Zoom/почты** (Экран 3) — «звонок → задачи» с провенансом и дедупом.
- **Конструктор + эскалация** (Экран 2) — самокоррекция дефолта роли.
- **Онбординг компании** (Экран 5) — ТЗ → штат из библиотеки.
- **Результат:** на экране видно то, чего нет у чат-ассистентов — штат, библиотека, разбор реальности.

## Что делать с KAM-демо (для Вячеслава)

`org-kam.js` (280 КБ), `kam-project.js`, `kam-assist.js`, `renderProject`/`renderRoadmap` — это **одна инстанциация** роли/компании, зашитая экранами.

- **Не выбрасывать** — это ценный проработанный пример.
- **Но:** позиционировать как *данные укомплектованной компании* поверх общей модели, а не как отдельный продукт. KAM-специфичные экраны (`project`, `roadmap`) — понизить до вложенных видов роли, не первый уровень.
- **Развилка (решение заказчика):** переносим KAM-компанию на новую модель высот **или** строим нейтральный пример, а KAM оставляем как есть до стабилизации модели. → см. открытый вопрос.

## Порядок для разработки (рекомендация)

1. **Фаза 0 + Фаза 1** вместе — это даёт видимый эффект «перегруз ушёл» за один заход.
2. Затем **Фаза 2** (грамматика) — дешёвая, большой эффект консистентности.
3. **Фаза 3** — по одному дифференциатору, начиная с **кандидатов из Zoom** (самое «вау» и ближе всего к готовому `kam-assist.js`).
