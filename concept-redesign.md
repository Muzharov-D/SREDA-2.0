## 1. ПУЛЬС → ИМПУЛЬС

### Что не работает сейчас

| Проблема | Почему это баг, не фича |
|----------|------------------------|
| Частица ≠ задача | Летит зелёный шарик «PR #483» — но это не PR, это декорация. Не ясно: кто пишет, где в pipeline, заблокирован ли гейтом |
| Кружочек ≠ команда | «Разработка · 52 чел.» — но в данных DEV_TEAM 27 человек, а в пульсе отдела 12 функций. Цифры висят в воздухе |
| Нет причинно-следственной связи | Частица прилетает в «Разработку» — и что? Что изменилось? Кто её поймал? |
| Кликабельность неочевидна | Кружок «Бэкенд · 3» — но это не три человека, это абстракция. Почему туда можно ткнуть — не ясно |

### Концепция: ЭКГ компании (или фМРТ организации)

Метафора не «планеты и орбиты», а **«нервная система, где каждый нейрон — живой сотрудник»**.

#### Нейроны (узлы)

| Визуал | Что это | Откуда данные |
|--------|---------|---------------|
| Круг с фото/аватаром + именем | Человек | DEV_TEAM, COCKPITS.team |
| Ромб/шестиугольник с эмодзи | Цифровой сотрудник | DIGITAL_STAFF |
| Размер нейрона | Количество задач в работе (загрузка) | board (backlog/work/review) |
| Цвет свечения | Статус: зелёный (в потоке), жёлтый (ждёт), красный (блокер/гейт) | stepGate, issue.sev |

#### Синапсы (связи)

| Визуал | Что это | Откуда данные |
|--------|---------|---------------|
| Линия между двумя нейронами | Передача работы | FLOWS.steps (who → next who) |
| Толщина линии | Объём передач за период | Счётчик handoff между парой |
| Цвет линии | Тип задачи | DEPT_TASK[role].c |
| Анимация «импульса» | Задача в пути. Не просто «летающий шарик», а **волна возбуждения**, которая запускает свечение следующего нейрона, когда доходит | board + flowState |

#### Почему это не банальные «нейросвязи»

Обычно нейросети в интерфейсах — красивые линии без смысла. Здесь:

1. **Каждый нейрон — реальный сотрудник из моков**. Ты видишь «Игорь · Тимлид», а не «Разработка · 52 чел.». Кликаешь — проваливаешься в его RPG-профиль.
2. **Каждый импульс — реальная задача из данных**. «PR #482 · Игорь → Аня» — и когда импульс доходит до Ани, её нейрон загорается, и появляется всплывашка: «Аня: ревьюит PR #482, 2 замечания».
3. **Гейт = тормозной синапс**. Если PR заблокирован sev1-риском, импульс **застревает** на синапсе и мигает красным. Ты видишь: сигнал идёт, но нейрон «Ира · Юрист» не возбуждается — потому что гейт закрыт.

#### 3 уровня зума

| Уровень | Нейроны = | Синапсы = | Данные |
|---------|-----------|-----------|--------|
| **Макро** | 8 крупных кластеров (отделы) | Толстые межотделовые связи = передачи между доменами | DEPARTMENTS + FLOWS |
| **Мезо** | Функции/подотделы | Задачи внутри отдела | COCKPITS.team.fn + board |
| **Микро** | Конкретные люди + их цифровые рои | Конкретные тикеты/PR/КП | DEV_TEAM + DIGITAL_STAFF + board |

#### Как это решает жалобы

- **Кто делает задачу**: подписано на импульсе — «👤 Игорь: ревью» или «🤖 Алгоритм: кодоген»
- **Кликабельность**: все нейроны с `cursor:pointer`, при наведении подсвечиваются все синапсы этого нейрона (видно, с кем он связан)
- **Числа совпадают**: нейроны = реальные люди из DEV_TEAM / DIGITAL_STAFF, не абстрактные «52 человека»

## 2. ПРОФИЛЬ СОТРУДНИКА КАК RPG

### Дисбаланс сейчас

| | Человек | Цифровой сотрудник |
|--|---------|---------------------|
| Данные | `{name, role, task, emoji, fn}` | `{name, title, fn, lead, model, status, now, ji: {mission, duties, limits, kpi, esc}}` |
| Что видно | Имя + задача | Полноценная должностная инструкция |
| Проблема | Бедный объект | Богатый объект — но продукт про людей |

## 3. РАСШИРЕННЫЙ ПРОФИЛЬ: что показывать при клике

### Для человека (полный экран)

**Левая колонка — RPG-карточка:**
- Аватар (фото или эмодзи), имя, роль, отдел, уровень сеньорности
- **Полоска HP** = загрузка (сколько задач в work + review)
- **Характеристики** = скиллы со звёздами (генерируются из роли + fn)
- **Активные квесты** = тикеты из board с прогресс-барами
  - PR #482: [██████░░░░] 60% · 🔴 sev1-риск
  - JIRA-1201: [████████░░] 80% · 🟡 в работе

**Правая колонка — контекст:**
- **Рой** (призванные цифровые): список с их статусами, моделями, текущими задачами
- **История передач**: с кем и что передавал за последние N дней (из FLOWS)
  - «09:35 → передал КП «Гамма» → Ира (Юр)»
  - «09:52 → принял Merge PR #482»
- **KPI**: персонализированные метрики из отделных метрик
  - «PR принято: 144 · Ревью с 1-й подачи: 86% · Эскалаций: 4%»
- **Должностная инструкция** (новое — аналог ji у цифровых):
  - Миссия: «Вести команду бэкенда, держать стандарты, эскалировать критичное»
  - Обязанности: ревью, планирование, менторство, онбординг
  - Границы: «Не пишет код в прод · Не меняет архитектуру без совета · Не принимает решения о найме/увольнении»
  - Эскалация: «Критичный риск → CTO · Конфликт интересов → HRD»

### Для цифрового (полный экран)

**Левая колонка — RPG-карточка:**
- Эмодзи-аватар, имя, титул, модель (Claude Sonnet), статус, стоимость вызова
- **Полоска MP** = задач в неделю / лимит
- **Характеристики** = скорость, качество, автономия, безопасность
- **Заклинания** = ji.duties со звёздами
- **Проклятия** = ji.limits (красным)

**Правая колонка — контекст:**
- **Руководитель-человек**: кто эскалирует (lead), его контакты, текущий статус
- **Активный квест**: `now` с прогрессом и артефактами
- **KPI**: ji.kpi в виде графиков/баров
- **История**: последние 5 задач, выполненных за неделю
- **Интеграции**: к каким системам подключен (WORKER_INTEGRATIONS) — CRM, репозиторий, CI/CD
- **Аудит**: кто и когда давал ему задачи (из auditLog)

## 4. СВОДКА: что нужно добавить в данные

Минимальный набор для реализации:

| Что добавить | Где | Зачем |
|--------------|-----|-------|
| `skills: [{name, level}]` | DEV_TEAM / COCKPITS.team | RPG-характеристики человека |
| `level: 1-5` | DEV_TEAM | «Lvl 5 · Тимлид» |
| `quests: [{id, title, progress, status}]` | board-данные | Активные квесты в профиле |
| `mission, duties, limits, esc` | Новое поле для человека | Должностная инструкция (аналог ji) |
| `achievements: []` | Калькуляция из KPI | Достижения в RPG-карточке |

**Важно**: всё это можно вывести из **существующих данных** — не нужно придумывать новые моки с нуля.

- Скиллы → из `role + fn`
- Квесты → из `board`
- Достижения → из `metrics`
- Нужно только **структурировать** и **визуализировать**
