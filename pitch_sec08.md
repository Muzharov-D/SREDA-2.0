# sec08: Архитектура: Фазовая модульность

## Пять функциональных блоков

Архитектура «Среды» построена не как монолитное приложение, а как **система из пяти функциональных блоков**, каждый из которых отвечает за строго определённый слой работы с AI-агентами. Такое разделение позволяет запускать платформу по частям — от минимального viable-продукта до полноценного enterprise-решения, при этом каждый блок развивается и масштабируется независимо от остальных.

**Framework** — ядро всей системы. Это runtime для AI-агентов: оркестрация задач, управление состоянием диалогов, маршрутизация вызовов между агентами, обработка ошибок и восстановление после сбоев. Framework отвечает за то, чтобы десятки или сотни агентов работали согласованно, не конфликтовали за ресурсы и не теряли контекст при переключении между задачами. Технически это распределённый orchestrator с поддержкой long-running процессов и event-driven архитектурой. Framework реализует паттерн «supervisor-workers»: центральный координатор распределяет задачи между специализированными агентами, отслеживает их выполнение и при необходимости перезапускает или перенаправляет запросы. Внутри Framework работает система памяти агентов — не просто история чатов, а структурированное хранилище фактов, выводов и промежуточных результатов, доступное для семантического поиска.

**Data Platform** — профиль пользователя и система обучения. Каждый сотрудник, каждый проект и каждый агент в «Среде» имеют цифровой профиль: навыки, предпочтения, историю взаимодействий, контекстные данные. Data Platform агрегирует эту информацию, строит векторные представления (embeddings) для семантического поиска и обеспечивает персонализацию ответов агентов. Платформа использует гибридные хранилища: реляционные базы для структурированных данных, векторные базы (типа Qdrant или Milvus) для семантического поиска, и объектные хранилища для документов и артефактов. Data Platform также управляет fine-tuning и адаптацией моделей под специфику конкретной компании — без необходимости делиться данными с внешними провайдерами. Система поддерживает инкрементальное обучение: по мере накопления корпоративных данных качество ответов агентов улучшается, а не деградирует.

**Dashboard** — аналитика и KPI. Здесь собираются метрики эффективности: сколько задач выполнено агентами, какова экономия времени, какие процессы стали узкими местами, где требуется человеческое вмешательство. Dashboard строит отчёты в реальном времени и позволяет руководству принимать решения на основе данных, а не интуиции. Система отслеживает ключевые метрики: task completion rate (доля задач, решённых без человека), human-in-the-loop frequency (частота эскалации к людям), cost per task (стоимость обработки одной задачи), model accuracy drift (деградация качества моделей со временем). Dashboard интегрируется с существующими BI-системами компании через стандартные API и коннекторы.

**Portal** — единая точка входа для найма, маркетплейса и биллинга. Через Portal компания подключает новых сотрудников и агентов, приобретает готовые решения из маркетплейса, управляет подписками и расходами. Это коммерческий слой платформы, который превращает «Среду» из внутреннего инструмента в экосистему с экономической моделью. Portal поддерживает ролевую модель: администраторы управляют лицензиями, team leads назначают агентов на проекты, сотрудники получают доступ к нужным инструментам. Маркетплейс внутри Portal позволяет покупать готовых агентов, шаблоны workflow и интеграции с внешними сервисами.

**Channels** — интерфейсы взаимодействия. Telegram, Slack, email, IDE, веб-чат и корпоративные мессенджеры. Агенты «Среды» не привязаны к единому интерфейсу: они работают там, где уже работают люди. Каждый канал адаптирует формат ответа под свою среду — кратко в Slack, развёрнуто в email, с кодом и подсветкой синтаксиса в IDE. Channels реализованы как адаптеры (adapters) с единым протоколом внутренней коммуникации: добавление нового канала требует только реализации адаптера, без изменения ядра системы.

Поверх всех пяти блоков лежит **слой Governance** — сквозная система управления: безопасность, контроль доступа, аудит, kill-switch. Она не является отдельным блоком в классическом смысле, а пронизывает всю архитектуру, применяя политики на каждом уровне. Governance реализует принцип zero-trust: ни один компонент не доверяет другому автоматически, каждое взаимодействие верифицируется.

## Как блоки комбинируются по фазам

Главное архитектурное решение «Среды» — **фазовая модульность**. Клиент не покупает «всё сразу», а подключает ровно те блоки, которые нужны для текущей фазы зрелости. При этом комбинация блоков меняется в зависимости от того, кто является техническим хозяином инфраструктуры.

**Фаза Inside** — полная платформа, все пять блоков, развёрнутая on-premise или в SaaS-режиме. Это «полная нервная система» компании: Framework управляет агентами, Data Platform хранит профили, Dashboard показывает метрики, Portal обеспечивает найм и закупки, Channels интегрируются с существующими коммуникациями. Governance работает на полную мощность. Клиент полностью контролирует инфраструктуру, данные не покидают периметр компании. Эта фаза подходит для enterprise-клиентов с собственными IT-отделами и жёсткими требованиями к compliance. В режиме on-premise «Среда» развёртывается в Kubernetes-кластере клиента, интегрируется с корпоративным Active Directory или LDAP, и использует внутренние хранилища данных. В SaaS-режиме мы предоставляем выделенный инстанс (single-tenant), а не shared infrastructure — это важное отличие от классических мультитенантных решений.

**Фаза Talent** — облегчённая конфигурация для подбора персонала. Здесь активны Portal, Data Platform, Channels и Dashboard (в режиме customer view). Framework скрыт за «Средой» — клиент не управляет агентами напрямую, а взаимодействует с готовыми AI-рекрутерами через привычные каналы. Data Platform строит профили кандидатов, анализируя резюме, портфолио и историю коммуникаций. Portal управляет воронкой найма: от публикации вакансий до оформления оффера. Channels обеспечивают коммуникацию — кандидаты могут отвечать в Telegram или email, не устанавливая новых приложений. Dashboard показывает HR-метрики: time-to-hire, conversion по этапам, качество подбора. Это «чёрный ящик» для бизнеса: работает без IT-интеграции, запускается за дни, а не месяцы. HR-менеджер получает готовый инструмент, а не платформу для настройки.

**Фаза Forge** — конфигурация для разработки и прототипирования. Portal, Framework, Data Platform, Channels и Dashboard работают на стороне «Среды». Клиент видит только «окно в кузницу» — веб-интерфейс или IDE-плагин, через который отправляет задачи агентам и получает результаты. Вся тяжёлая инфраструктура — runtime, GPU-кластеры, хранилища, шлюзы LLM — находится в облаке «Среды». Клиент платит за использование, а не за содержание инфраструктуры. Это идеальная точка входа для стартапов и небольших команд, которым нужна мощь enterprise-платформы без капитальных затрат. Forge поддерживает совместную разработку: несколько разработчиков работают с одним агентом, видят его действия в реальном времени, могут прерывать или корректировать поведение. Агенты в Forge имеют доступ к репозиториям кода (через read-only интеграции), документации и тестовым средам.

Ключевое преимущество такого подхода — **единая кодовая база для всех фаз**. Мы не разрабатываем три разных продукта. Мы разрабатываем один продукт с пятью блоками и тремя режимами развёртывания. Это сокращает time-to-market, упрощает поддержку и позволяет клиенту «расти внутри» платформы, переходя от Forge к Inside без миграции данных. Когда клиент решает перейти с Forge на Inside, мы переносим его Data Platform и настроенных агентов в его инфраструктуру — процесс занимает часы, а не недели.

## LLM Gateway: шлюз языковых моделей

**Провайдер-агностичность** — фундаментальное решение архитектуры «Среды». Мы не привязаны ни к одному вендору LLM, что даёт три стратегических преимущества: независимость от ценовой политики, доступ к лучшим моделям для каждой задачи и защита от vendor lock-in. Если OpenAI повышает цены или меняет политику использования, мы переключаемся на Anthropic, Google или локальные модели — без изменения бизнес-логики клиента.

Технически это реализовано через **LLM Gateway** — интеллектуальный маршрутизатор запросов к языковым моделям. Gateway интегрирован с OpenRouter, что обеспечивает доступ к более чем 200 моделям от десятков провайдеров: OpenAI, Anthropic, Google, Meta, Mistral, Cohere и множества специализированных сервисов. Для каждой задачи Gateway выбирает оптимальную модель на основе критериев: требуемое качество, допустимая задержка (latency), стоимость токена, наличие специализированных навыков (например, код vs аналитика vs креатив). Gateway поддерживает A/B-тестирование моделей: можно направлять 10% трафика на новую модель и сравнивать качество ответов с текущей, прежде чем делать полный переход.

Параллельно Gateway поддерживает **локальные open-weight модели** — Qwen, DeepSeek, Llama и другие — через inference-движки vLLM и SGLang. vLLM обеспечивает высокую пропускную способность за счёт PagedAttention — техники, которая эффективно управляет памятью GPU при обработке запросов переменной длины. SGLang оптимизирован для сложных структурированных выходов и работы с функциями (function calling). Это критически важно для клиентов с требованиями data sovereignty: чувствительные данные обрабатываются локально, без отправки во внешние API. Локальные модели развёртываются в том же кластере, что и Framework, с автоматическим масштабированием GPU-ресурсов под нагрузку. При росте числа запросов система автоматически добавляет inference-ноды; при снижении — освобождает ресурсы.

Gateway реализует **цепочки отказоустойчивости (fallback chains)**. Если основная модель недоступна, перегружена или вернула ошибку, запрос автоматически перенаправляется на резервную модель с сохранением контекста. Для критичных задач можно настроить каскад из трёх-четырёх моделей разных провайдеров. Например: сначала GPT-4, при недоступности — Claude 3, при недоступности — локальная DeepSeek. Контекст (история диалога, системные инструкции) сохраняется при каждом переключении.

**Prompt caching** — ещё один механизм оптимизации. Gateway распознаёт повторяющиеся префиксы контекста (системные инструкции, базовые знания компании, описания инструментов) и кэширует их на уровне инфраструктуры, избегая повторной отправки в API. Это сокращает расходы на 30–60% для типовых рабочих сценариев, где системный промпт занимает значительную часть контекстного окна. Кэширование работает как для внешних API (OpenRouter поддерживает prompt caching для совместимых моделей), так и для локальных моделей.

Наконец, **per-task routing к самой дешёвой способной модели**. Gateway анализирует сложность задачи и маршрутизирует простые запросы к лёгким моделям (стоимость — доли цента), а сложные — к мощным (стоимость — единицы центов). Это не жёсткое правило, а обучаемая политика: система собирает feedback loop по качеству ответов и корректирует маршрутизацию. Со временем Gateway «учится» распознавать паттерны запросов и автоматически выбирать оптимальную модель без участия человека.

## Архитектура безопасности

Безопасность в «Среде» — не дополнительная опция, а **архитектурный слой, встроенный в фундамент**. Каждый компонент, каждый вызов, каждый байт данных проходит через систему контроля. Мы проектируем под угрозу, а не добавляем защиту постфактум.

**Sandbox-изоляция** — первый рубеж защиты. Каждый агент выполняется в изолированной среде: gVisor (userspace-ядро с ограниченным доступом к системным вызовам), Kata Containers (контейнеры с собственным минимальным ядром) или Firecracker (microVM от AWS, оптимизированная для быстрого запуска). Это означает, что даже если агент скомпрометирован — например, через prompt injection заставили выполнить вредоносный код — он не может выйти за пределы своей песочницы, получить доступ к данным других агентов или к хост-системе. Изоляция применяется не только к агентам, но и к пользовательскому коду, который агенты могут генерировать и выполнять. Каждый запуск кода — это отдельная microVM с ограниченным временем жизни, без сетевого доступа и с ограниченными ресурсами CPU/памяти.

**Разделение Data Plane и Control Plane** — классическое сетевое решение, адаптированное под AI-инфраструктуру. Data Plane обрабатывает собственно данные: запросы к LLM, профили пользователей, результаты работы агентов, векторные embeddings. Control Plane управляет конфигурацией, политиками, маршрутизацией, масштабированием, обновлениями. Эти два плана физически разделены: они работают в разных namespace Kubernetes, используют разные сетевые сегменты, разные учётные записи. Компрометация Data Plane не даёт доступа к управлению инфраструктурой, а компрометация Control Plane не открывает данные. Это стандартный подход в телекоммуникациях и облачных провайдерах, который мы переносим в мир AI-агентов.

**SPIFFE/SPIRE** — система идентификации сервисов. Вместо статических API-ключей и паролей, которые могут утекать или быть скомпрометированы, каждый компонент «Среды» получает краткосрочный криптографический идентификатор, привязанный к его роли и окружению. Агент A может доказать, что он именно агент A, работающий в конкретном namespace конкретного кластера — и только тогда получить доступ к нужным ресурсам. Идентификаторы автоматически обновляются каждые несколько часов, устраняя риски, связанные с долгоживущими credentials. Это устраняет целый класс атак с компрометацией статических ключей.

**PEP (Policy Enforcement Point)** — точка принудительного применения политик на каждом действии. Каждый вызов агента, каждый доступ к данным, каждая операция с внешним API проходит через PEP, который проверяет: имеет ли инициатор право на это действие? Соответствует ли действие текущим политикам компании? Не нарушает ли оно регуляторные ограничения (например, GDPR или 152-ФЗ)? PEP работает в реальном времени, с задержкой менее миллисекунды, и логирует каждое решение для последующего аудита. Политики описываются на декларативном языке и могут обновляться без перезапуска системы.

**JIT-доступ (Just-In-Time)** — временное предоставление прав. Вместо постоянных привилегий агент получает разрешение на конкретное действие в конкретный момент, с автоматическим отзывом по завершении. Если агенту нужно отправить email, он получает право на одну отправку — и только на неё. Если нужно прочитать файл — право на чтение конкретного файла в течение 5 минут. Это минимизирует blast radius при компрометации: даже если агент захвачен злоумышленником, он не может использовать накопленные привилегии для дальнейшего продвижения.

**Instant kill-switch** — мгновенная остановка. В любой момент администратор или автоматизированная система мониторинга может остановить любого агента, любой кластер или всю платформу целиком. Kill-switch работает на уровне Control Plane и обходит обычные механизмы авторизации — это аварийный тормоз, а не политика. При срабатывании kill-switch все выполняющиеся задачи немедленно прерываются, агенты переводятся в режим ожидания, все внешние вызовы блокируются. Восстановление требует явного действия администратора с двухфакторной аутентификацией.

**Audit trail to SIEM** — полная аудиторская тропа. Каждое действие логируется с неизменяемой записью: кто (или что) инициировал, когда, с какого IP/хоста, какой результат, какие данные были затронуты, какие политики применялись. Логи в реальном времени отправляются в SIEM-системы клиента (Splunk, Elastic, Sentinel, ArcSight) для корреляции с другими событиями безопасности. Формат логов соответствует стандартам CEF и LEEF, что обеспечивает совместимость с большинством enterprise-SIEM без дополнительной трансформации.

**Air-gapped option** — возможность полной изоляции. Для клиентов с максимальными требованиями к безопасности — государственных структур, оборонных предприятий, финансовых регуляторов — «Среда» может работать в полностью изолированной среде: без интернет-доступа, с локальными LLM, с физическим разделением сетей. Все обновления доставляются через защищённые каналы (физические носители или выделенные линии), все компоненты верифицируются криптографически перед установкой. Air-gapped развёртывание включает полный набор функциональности, за исключением внешних LLM-провайдеров — вместо них используются локальные модели, развёрнутые в том же периметре.

## Модульность как стратегия

Модульность «Среды» — это не просто техническое удобство, а **бизнес-стратегия масштабирования**, которая определяет, как мы выходим на рынок и как растём вместе с клиентами.

Монолитные платформы требуют от клиента «всё или ничего»: купи лицензию на весь продукт, разверни всю инфраструктуру, обучи весь персонал, перестрой процессы под единый интерфейс. Это отсекает стартапы и средний бизнес, для которых такие вложения неоправданны. Enterprise-клиенты тоже сопротивляются: они не хотят менять работающие системы ради одного вендора. «Среда» работает иначе — мы приходим туда, где уже работают клиенты, и предлагаем ровно тот объём функциональности, который нужен сейчас.

Клиент начинает с одной фазы — обычно с Forge для разработки или Talent для HR — и подключает остальные по мере роста. Каждый блок можно приобрести отдельно: нужен только Dashboard для аналитики существующих агентов — пожалуйста. Нужен только Portal для внутреннего маркетплейса AI-инструментов — возможно. Нужен Framework для интеграции с собственной data platform — легко. Такая гранулярность даёт три стратегических эффекта.

**Во-первых**, снижается порог входа. Клиент платит за то, что использует, а не за потенциальные возможности. Стартап может начать с Forge за сотни долларов в месяц, а не с enterprise-лицензии за десятки тысяч. По мере роста команды и сложности задач расходы растут пропорционально ценности, которую платформа создаёт.

**Во-вторых**, ускоряется цикл принятия решения. Согласовать покупку одного блока для пилотного проекта проще, чем enterprise-лицензию на всю платформу. Procurement проходит быстрее, юридическая проверка проще, бюджетные риски ниже. Клиент получает возможность «попробовать» до «купить» — и мы получаем возможность доказать ценность на практике, а не в презентации.

**В-третьих**, создаётся естественная воронка upsell. Клиент, начавший с Talent, со временем вырастает до Inside: HR-отдел показывает эффективность, другие департаменты хотят такой же инструмент, и компания расширяет использование «Среды». Каждый новый блок добавляется без миграции данных и переобучения — это upsell без friction. Мы масштабируем выручку с одного клиента, не увеличивая затраты на привлечение.

Это и есть путь «Среды» от стартапа до enterprise: не через переделку продукта под каждый сегмент, а через комбинаторику одних и тех надёжных блоков. Мы строим не монолит, который нужно продавать целиком, а экосистему, которая растёт вместе с клиентом — от первого AI-агента до корпоративной нервной системы.
